Research brief · 12/07/2026

Medical QA, grounded in text and graph.

Input là câu hỏi text. Output là câu trả lời text có dẫn chứng. Nghiên cứu không cố chứng minh Graph-RAG luôn tốt hơn; nó đo chính xác graph giúp loại câu hỏi nào khi so với một Text-RAG đủ mạnh.

01 · SCOPE

Bài toán và ranh giới

Report pháp luật chỉ cung cấp tư duy baseline → module → ablation. Domain, schema, dữ liệu và metric được thiết kế lại cho medical.

Input

Medical question

Why should propranolol generally be avoided in patients with asthma?
→
Output

Evidence-grounded answer

Propranolol blocks β2 receptors and may cause bronchoconstriction… Sources: PMID… · PrimeKG path…
Primary track

BioASQ long-form QA

Đánh giá chatbot text-to-text và retrieval nhờ ideal answer, PubMed document IDs và gold snippets.

Graph component track

PrimeKGQA

Đánh giá entity/relation linking, graph traversal và KG reasoning bằng question, answer và SPARQL sinh từ PrimeKG; báo riêng 2/3/4-node complexity.

Research question

PrimeKG có thêm giá trị ngoài BM25 + MedCPT + reranker không?

Expected nuance

Graph có thể giúp drug–disease, contraindication và multi-hop; không chắc giúp literature-specific QA.

Responsible boundary

Prototype nghiên cứu, không thay bác sĩ, không dùng để đưa quyết định điều trị.

EDA bắt buộc

Ghi số record/split; null/duplicate; phân phối độ dài title/abstract/question; question type; số gold PMID/snippet mỗi câu; tỷ lệ gold PMID tồn tại trong corpus; entity type và PrimeKG coverage. Xuất bảng + histogram + manifest hash.

Không có bước crawl

Chỉ tải dataset public, pin revision và kiểm tra license/checksum. Nếu một field/split khác dataset card, cập nhật schema manifest trước khi code; tuyệt đối không tự suy đoán tên cột trong evaluator.

Chốt: không phải “chỉ bấm test”. Cần khoảng 1–2 tuần dựng integration/evaluator; sau đó phần lớn thời gian là thí nghiệm có kiểm soát.
03 · MODELS

Model/API nào dùng, dùng để làm gì

API chỉ đảm nhiệm sinh câu trả lời và hỗ trợ chấm. Retrieval biomedical vẫn dùng các model open-weight chạy local. Không thay model giữa hai pipeline đang được so sánh.

Cấu hình đã xác minh 13/07/2026: dùng deepseek-v3.2 làm generator cho toàn bộ B0–G2; dùng cerebras/gpt-oss-120b làm LLM judge độc lập qua gateway OpenAI-compatible. Model-list, JSON/citation smoke và calibration dev đều có artifact. Không trộn model trong cùng paired run.

Fallback rõ ràng

Không có fallback tự động trong paired run. Nếu DeepSeek lỗi/quá quota thì dừng run, ghi lỗi và chọn Qwen một lần trước khi chạy lại toàn bộ population với run ID/model hash mới. Judge lỗi cũng dừng; không đổi judge sau khi xem kết quả.

Không gửi dữ liệu bệnh nhân

Dataset benchmark là dữ liệu nghiên cứu công khai. Không gửi EHR/PII/PHI lên free-tier API. Cache theo hash của prompt + evidence; log token, latency, retry và lỗi 429, tuyệt đối không log API key.

05 · RAG SYSTEM

Kiến trúc end-to-end: offline → online → evaluation

Offline chuẩn bị index một lần. Online nhận question và trả answer. Evaluation replay đúng pipeline online trên question IDs đã khóa; gold labels không bao giờ đi vào retriever hoặc prompt sinh đáp án.

High-level architecture

OFFLINE BUILD (chạy khi data/config đổi)
BioASQ text-corpus → validate/EDA → C0/C2 chunks ─┬→ BM25 index
                                                     └→ MedCPT Article Encoder → FAISS
PrimeKG nodes/edges → normalize aliases/types → adjacency + provenance index
PrimeKGQA → validate answer/SPARQL → RDF/version compatibility smoke

ONLINE QA (chạy cho mỗi câu hỏi)
Question → normalize + medical entity detection ─┬→ Q0/Q1/Q2 query
                                                  │    → BM25 + MedCPT
                                                  │    → RRF → cross-encoder → text evidence
                                                  └→ entity linking → PrimeKG 1–2 hop
                                                       → path filter/rank → graph evidence
Text evidence + graph evidence → fixed-budget fusion → citation IDs
→ frozen LLM API inference → citation validation → answer + evidence + uncertainty

EVALUATION (không can thiệp vào online answer)
Frozen QA IDs + gold PMID/snippets/ideal answer → shared harness
→ retrieval + entity/path + answer/citation + latency/cost metrics
→ paired statistics + blind human audit + error taxonomy

Query strategies

Q0 → Q1; Q2 exploratory

Q0 giữ nguyên question. Q1 thêm normalized medical entities/aliases. Q2 tạo tối đa ba search queries. Gold miss chỉ dùng chọn failure subset để phân tích; runtime Q2 phải dùng confidence/disagreement rule tune trên dev rồi freeze.

Chunk strategies

C0 và C2 trên cả sparse/dense

C0: một PubMed abstract. C2: sentence-aware tối đa 256 tokens. Chạy cả hai với BM25 và MedCPT để tránh chọn chunk thiên vị BM25; mọi chunk giữ parent PMID và offset.

Graph routing

Không ép graph cho mọi câu

G2 mặc định thử graph nhưng chỉ đưa path đạt validity/relevance threshold vào context; thiếu path thì trả budget về text. Exploratory router chỉ kích hoạt graph khi có linked entities và relation intent phù hợp. Oracle graph-answerability chỉ phân tích trần, không dùng khi inference.

Answer contract

Text output nhưng có provenance

Người dùng nhận answer tự nhiên, PubMed citations, PrimeKG paths và trạng thái evidence. PrimeKG relation là association trừ khi source định nghĩa khác; hệ thống phải abstain khi evidence không đủ, không biến graph edge thành khẳng định nhân quả.

Phase 1 · Need

RAG có cần không?

B0 → B1/B2. Nếu retrieval không cải thiện answer hoặc citation, kiểm tra corpus/retriever trước khi thêm graph.

Phase 2 · Strength

Text baseline đã đủ mạnh?

B1/B2 → B3. BM25 xử lý từ khóa; MedCPT xử lý semantic gap; reranker kiểm tra lỗi xếp hạng.

Phase 3 · Structure

Graph thêm gì?

B3 ↔ G1 ↔ G2. Chỉ quy lợi ích cho graph khi G2 thắng B3 dưới cùng generator và evidence budget.

Phase 4 · Causality

Vì sao graph giúp?

Ablation. 1/2-hop, random neighbors, path reranker và conditional routing tách coverage khỏi reasoning thật.

Phase 5 · Trade-off

Có đáng triển khai?

So accuracy/citation với latency, token và cost. Module tăng điểm rất nhỏ nhưng tăng chi phí lớn sẽ bị loại.

Stop rule

Không test vô hạn

B3/G2 bắt buộc, không dừng theo interim score. Chỉ báo graph feasibility failure nếu entity coverage <20% hoặc cap 200 candidates/seed bị chạm ở >25% dev questions; optional ablations chỉ chạy sau core.

100 BioASQ questions simple-random, seed 20260712. Đây là estimand human chính cho correctness và graph usefulness; lưu IDs + inclusion probability + SHA-256 trước khi xem output.

Manual review

Hai reviewer độc lập, blind pipeline, randomize B3/G2 order và adjudicate bất đồng. PrimeKGQA thay phần lớn việc tự tạo gold graph paths; human audit chỉ kiểm tra usefulness trên câu hỏi BioASQ thực tế.

Graph labels frozen

answerability: yes/partial/no · reasoning: single fact/one-hop/multi-hop/literature/list/other · relation: drug–disease/adverse effect/contraindication/symptom/gene/treatment.

Failure policy

API failure sau 3 retry vẫn giữ record và correctness=0; invalid PrimeKGQA answer/query là failure; missing citation cho major claim cho recall 0; exclusion chỉ theo rule độc lập pipeline.

Frozen LLM judge contract · scale-up scorer có điều kiện

population         every frozen locked-test question ID; missing output creates
                   a retained failure record and correctness=0, never excluded
judge             gateway model ID cerebras/gpt-oss-120b; verified 13/07/2026
                  must differ from generator; unavailable → freeze replacement before test
correctness pass  question + reference answer + candidate answer; NO evidence

hidden            pipeline name, retrieval scores, competing answer, experiment outcome
output JSON       score after evidence-based justification · claim labels · confidence
decoding           temperature=0 · frozen prompt/config · invalid JSON retry≤2

estimand          human random-sample paired mean is primary regardless of gate;
                  full-set LLM judge is confirmatory only after calibration passes
statistics        bootstrap at question level; claims are not independent samples

PrimeKGQA scoring contract

compatibility      pin dataset files/MD5 + matching PrimeKG RDF IRI/predicate schema
smoke gate         execute 100 gold SPARQL; executable rate must be ≥99%
answer normalize   canonical resource IDs; set semantics; preserve node/relation type
primary            answer-set exact match + precision/recall/F1
query              executable-query rate + execution accuracy
secondary          entity accuracy · relation accuracy · graph-pattern overlap
strata             2-node · 3-node · 4-node; seen/unseen entity-relation patterns
forbidden          exact SPARQL string match as primary (equivalent queries may differ)
locked             official final_test.json once; validation only for thresholds

Correctness đo factual/medical validity: 2 không có lỗi quan trọng; 1 lỗi nhỏ nhưng kết luận chính đúng; 0 kết luận sai/lỗi trọng yếu/failure. Completeness đo riêng mức bao phủ essential points. Faithfulness = factual claims supported hoặc partially supported / tổng factual claims; contradictory và unsupported không được tính. PrimeKGQA release v1 ghi 83.999 pairs trong Zenodo/conclusion trong khi một bảng thống kê của paper cộng thành 85.368; manifest phải lấy count thật từ ba JSON files và báo discrepancy, không chép số marketing.

12 · DECISION

Kết quả nào cũng có giá trị nếu protocol đúng

Không đặt trước kết luận Graph-RAG thắng. Nghiên cứu tốt có thể chứng minh graph chỉ hữu ích có điều kiện.

Giả thuyết cần bác bỏ hoặc xác nhận: Graph có thể hữu ích khi entity linking đáng tin, PrimeKG có coverage và câu hỏi cần relational/multi-hop evidence; Text-RAG có thể phù hợp hơn với literature-specific QA.
